Data Modeling

Medallion Architecture

Six BigQuery datasets — each layer has a clear responsibility.

raw CSV from GCS · explicit schemas · ingestion metadata (_ingested_at, _batch_id) 9 tables
staging Clean names · cast types · deduplicate · null-safe · audit columns 9 views
intermediate Enriched joins · revenue calcs · customer summaries · campaign attribution 5 views
analytics Star schema · partitioned facts · clustered · surrogate keys · incremental merge 8 tables
reporting Business-ready marts for dashboards — daily sales, customer 360, ROI, retention 6 tables
snapshots SCD Type 2 history for customers & products (timestamp strategy) 2 snaps

dbt Engineering

dbt Best Practices in Action

✅ Testing

unique, not_null, relationships, accepted_values + 4 custom SQL tests for business rules

📸 Snapshots

SCD Type 2 for customers (city, status) and products (price, category) changes

📖 Documentation

schema.yml for every layer — sources, staging, intermediate, core, reporting

🔄 Incremental

fct_order_items & fct_web_events use merge strategy with configurable lookback

Source freshness checks warn at 24h, error at 48h on _ingested_at

🏷️ Macros

generate_schema_name, audit_columns, limit_data_in_dev for env-aware logic

Interview Q&A

Why medallion architecture?
Clear separation: raw preservation, cleaned staging, reusable logic, business-ready marts. Each layer tested independently.
Why partition + cluster?
Partitioning by date reduces scan cost. Clustering on customer_id/product_id speeds filters within partitions.
Incremental vs full refresh?
Incremental for large append-heavy facts. Full refresh for small dimensions or logic changes requiring history rebuild.
